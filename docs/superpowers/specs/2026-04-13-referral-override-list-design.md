# Referral Override List Design

## Goal

优化管理端用户编辑弹窗中的“邀请人返佣覆盖”区域，使其从当前“订阅返佣分组卡片”改为“可扩展的覆盖项列表”。

本次目标是：

- 将当前按分组卡片渲染的订阅返佣覆盖，重构为可增减的覆盖项列表
- 每条覆盖项显式包含：
  - `返佣类型`
  - `分组`
  - `覆盖总返佣率`
- 当前先支持 `订阅返佣` 一种类型
- 前端状态模型预留未来扩展到其他返佣类型的能力
- 继续复用现有订阅返佣管理接口，不在本次新增后端协议
- 优化空状态、重复项限制、行内错误反馈和草稿项管理

## Context

当前实现位于：

- `web/src/components/table/users/modals/SubscriptionReferralOverrideSection.jsx`

当前问题：

- UI 仍然以“订阅返佣按分组卡片”思路组织
- 每个分组都渲染一整张卡片，不适合“新增一条覆盖对象 / 删除一条覆盖对象”的管理方式
- 字段结构绑定在“订阅返佣”单一路径上，不利于未来新增其他返佣类型
- “当前生效总返佣率”占据较强视觉权重，但它在此处是辅助信息，不是核心编辑字段

用户已明确：

- “邀请人返佣覆盖”下面当前是订阅返佣
- 后续可能还会有别的返佣类型
- 本次应直接采用可扩展的列表模型，而不是继续做订阅返佣专用卡片

## Confirmed Decisions

以下内容已与用户确认：

- 页面结构采用“覆盖项列表”而不是“分组目录”
- 基线状态采用空列表加 `新增覆盖` 按钮
- 每条覆盖项都直接包含 `返佣类型`
- 当前类型只有 `订阅返佣`，但前端结构必须支持未来扩展
- 保存与删除继续沿用现有订阅返佣接口
- 列表保持轻编辑状态，不引入额外“查看态 / 编辑态”切换

## Out of Scope

本次不做：

- 新增第二种真实返佣类型的后端接口
- 改造订阅返佣后端返回结构
- 改造用户端邀请页 `InvitationCard` 的交互结构
- 为不同返佣类型设计不同的复杂字段集合
- 做跨类型的统一后端覆盖表

## Approaches Considered

### Approach A: 平铺覆盖项列表（采用）

结构：

- 顶部标题与说明
- 空状态或覆盖项列表
- `新增覆盖` 按钮
- 每行包含 `返佣类型`、`分组`、`覆盖总返佣率`、行操作

优点：

- 最符合“覆盖项”的语义
- 支持快速新增、删除、修改
- 为未来增加返佣类型预留最自然的扩展边界
- 在用户编辑 SideSheet 中占用纵向空间更可控

### Approach B: 按返佣类型分组的多节卡片

结构：

- 每种返佣类型一个 section
- section 下再列出各自覆盖项

问题：

- 当前只有一种类型时显得冗余
- 仍然保留“类型外层容器 + 内层覆盖项”的双层结构
- 对本次问题的直接收益小于实现复杂度

### Approach C: 保留卡片式展开编辑

结构：

- 每个覆盖对象仍保留信息卡
- 点击进入展开态编辑

问题：

- 更像信息展示，不像配置器
- 不利于连续新增和批量调整
- 与“支持增减对象”的需求不一致

## Decision

采用 **Approach A**。

原因：

- 这是唯一同时满足“当前是订阅返佣配置器”和“未来支持多种返佣类型”的最小重构
- 它直接把交互中心放到“覆盖对象”本身，而不是分组卡片
- 在不改后端协议的前提下，前端可以先完成结构升级

## Target UX

### Section Structure

“邀请人返佣覆盖”区域调整为：

- 标题：`邀请人返佣覆盖`
- 说明：`暂无覆盖时使用默认返佣规则`
- 主体：覆盖项列表
- 操作：顶部或列表尾部提供 `新增覆盖`

### Row Structure

每条覆盖项固定包含以下字段：

1. `返佣类型`
2. `分组`
3. `覆盖总返佣率`
4. 行操作

当前字段表现建议：

- `返佣类型`
  - 使用下拉选择
  - 本次只有一个选项：`订阅返佣`
  - 新增草稿行时默认选中该值
- `分组`
  - 使用下拉选择
  - 只显示当前订阅返佣可配置分组
- `覆盖总返佣率`
  - 使用百分比输入框
  - 范围 `0 ~ 100`
  - 保留两位小数
- 行操作
  - 已保存行：`保存`、`删除`
  - 未保存草稿：`保存`、`取消`

### Secondary Information

`当前生效总返佣率` 不再作为主视觉块展示。

如果需要保留，应仅作为弱化辅助文案出现在行内，例如：

- `当前默认总返佣率 45%`

目标是降低干扰，让主要视觉焦点回到“类型 / 分组 / 覆盖值”三个主字段。

### Empty State

当当前用户没有任何覆盖项时：

- 显示空状态文案：`暂无覆盖项，未设置时使用默认返佣规则`
- 展示 `新增覆盖` 按钮

空状态不再展示伪造卡片或不可编辑占位卡。

## Data Model

前端内部状态升级为通用覆盖项模型：

```js
{
  id,
  type,
  group,
  overrideRatePercent,
  effectiveTotalRatePercent,
  hasOverride,
  isDraft
}
```

字段含义：

- `id`
  - 前端本地唯一标识
  - 已保存项可由 `type + group` 派生
  - 草稿项使用临时 id
- `type`
  - 当前固定为 `subscription`
  - 未来新增其他返佣类型时复用此字段
- `group`
  - 当前返佣类型作用域下的分组
- `overrideRatePercent`
  - 可编辑的覆盖总返佣率
- `effectiveTotalRatePercent`
  - 当前默认或生效总返佣率，仅用于提示
- `hasOverride`
  - 表示当前行是否已经持久化为覆盖项
- `isDraft`
  - 表示当前行是否是尚未保存的前端草稿

## API Compatibility Strategy

本次不修改后端接口，仅增加前端适配层。

### Load

加载用户订阅返佣覆盖时：

- 继续请求现有接口：
  - `GET /api/subscription/admin/referral/users/:id`
- 将返回的 `groups[]` 映射为覆盖项列表
- 所有映射后的项统一标记为：
  - `type = subscription`

### Save

保存某一行时：

- 若 `type === subscription`
  - 调用：
    - `PUT /api/subscription/admin/referral/users/:id`
  - 请求体保持现有结构：

```json
{
  "group": "vip",
  "total_rate_bps": 4500
}
```

### Delete

删除某一行时：

- 若为已保存项，且 `type === subscription`
  - 调用：
    - `DELETE /api/subscription/admin/referral/users/:id?group=vip`
- 若为未保存草稿
  - 仅在前端移除，不发请求

### Future Extension

未来新增其他返佣类型时：

- 新增类型选项
- 在前端保存/删除适配层按 `type` 分流到不同接口
- 无需重写列表组件的交互骨架

## Validation and UX Rules

### Field Rules

1. `返佣类型` 必填
2. `分组` 必填
3. `覆盖总返佣率` 必填
4. `覆盖总返佣率` 范围为 `0 ~ 100`
5. 保留两位小数

### Duplicate Prevention

同一个 `返佣类型 + 分组` 组合不能重复创建。

实现建议：

- 当列表中已存在某个组合时，在新增草稿的分组下拉中禁用对应分组
- 或在保存时阻止并给出明确错误

优先使用“前置禁用 + 保存兜底校验”的双层策略。

### Row-Level Feedback

- 每条行独立保存
- 每条行独立删除
- 某一行失败时，不影响其他行
- 后端报错直接显示在对应行附近

### Editing Mode

列表维持“轻编辑”状态：

- 不再设计额外的整行展开/折叠编辑模式
- 已有项可直接调整并保存
- 草稿项可直接填写并保存

这样更符合配置器场景，也更适合在 SideSheet 中快速连续操作。

## Rendering and Interaction Notes

### Initial State

- 若存在已有覆盖项，直接渲染这些项
- 若无覆盖项，渲染空状态

### Add

- 点击 `新增覆盖`
- 插入一条草稿行
- 若当前仅有一个返佣类型，则默认填入 `订阅返佣`

### Cancel Draft

- 草稿行支持 `取消`
- 点击后仅移除该行，不触发接口

### Save Existing Row

- 已保存行点击 `保存`
- 触发对应类型的更新请求
- 保存成功后刷新该行数据，保持在当前列表中

### Delete Existing Row

- 已保存行点击 `删除`
- 触发对应类型的删除请求
- 删除成功后从列表中移除

## Implementation Notes

建议保持改动集中在以下前端范围：

- `web/src/components/table/users/modals/SubscriptionReferralOverrideSection.jsx`
- `web/src/helpers/subscriptionReferral.js`
- 对应组件测试文件

如需补充通用覆盖项 helper，应优先放在现有 `subscriptionReferral.js` 中，避免过早拆出新的抽象层。

## Risks

### Risk 1: UI 已可扩展，但后端仍是单类型

这是本次有意接受的限制。

处理方式：

- 明确把“多类型”收敛为前端结构预留，而不是承诺真实多类型功能已经完成

### Risk 2: 重复项规则只做前端约束

处理方式：

- 前端禁用重复组合
- 保存失败时继续承接后端错误，避免静默不一致

### Risk 3: 当前 helper 命名仍偏订阅返佣

处理方式：

- 本次只做必要重构
- 待未来第二种返佣类型出现后，再评估是否抽出更通用的 helper 命名空间

## Rollout

实施顺序建议：

1. 先补 helper 和组件测试，定义新的列表状态模型
2. 再重构 `SubscriptionReferralOverrideSection.jsx`
3. 本地验证新增、保存、删除、空状态
4. 验证旧订阅返佣覆盖数据的加载与编辑回归
