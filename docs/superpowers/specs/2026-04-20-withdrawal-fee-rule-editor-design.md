# Withdrawal Fee Rule Editor Design

## Goal

把当前后台“提现手续费规则”的纯 JSON 配置方式升级为可视化分段规则编辑器，降低运营配置门槛，并让用户侧手续费说明与试算结果更直观、更一致。

本次目标是：

- 后台不再让管理员直接手写 `WithdrawalFeeRules` JSON
- 改为“规则表格 + 行内展开编辑”的通用分段手续费编辑器
- 支持管理员任意新增、删除、调整区间和费率
- 统一手续费区间语义为“左开右闭”
- 保存时仍兼容后端现有 `WithdrawalFeeRules` 配置格式
- 用户提现弹窗展示自然语言规则说明与实时试算
- 对重叠区间、无效边界、无命中金额等场景提供明确校验和提示

## Context

项目当前已经具备以下基础：

1. 提现基础能力已上线
   - 用户可在钱包管理中发起提现
   - 管理员可审核、驳回、确认打款
   - 手续费规则已支持按金额区间配置多档

2. 当前手续费配置方式门槛较高
   - 后台 `提现手续费规则` 仍是一个 JSON 文本框
   - 管理员需要理解 `min_amount`、`max_amount`、`fee_type`、`sort_order` 等底层字段
   - 一旦区间或边界写错，配置很难被及时发现

3. 当前规则引擎语义不够贴近业务表达
   - 现有前后端逻辑按 `[min_amount, max_amount)` 理解区间
   - 业务期望的表达更接近：
     - `0 < amount <= 100`
     - `100 < amount <= 500`
     - `500 < amount <= 2000`
     - `amount > 2000`

4. 用户侧目前已有基础试算能力
   - 用户输入提现金额后，前端会显示手续费和到账金额
   - 但命中规则的展示仍偏底层，不够直观

## Problem Statement

当前方案的主要问题：

- 运营需要直接编辑 JSON，学习成本高
- 区间边界和排序依赖人工理解，不直观
- 错误配置只能在保存时报错，缺少结构化编辑体验
- 用户侧虽有手续费预览，但缺少清晰的规则说明
- 运营表达和系统内部表达没有充分分层

## Confirmed Product Decisions

以下内容已经和用户确认：

- 本次采用“通用分段表格编辑器”，不是只内置 4 档固定模板
- 管理员可以自由新增、删除、调整手续费规则
- 编辑方式采用“表格行内展开编辑”
- 暂不使用拖拽排序，先采用 `上移 / 下移`
- 统一使用“左开右闭”区间语义：
  - 第一档：`0 < 金额 <= X`
  - 中间档：`A < 金额 <= B`
  - 无上限档：`金额 > A`
- 区间重叠时禁止保存
- 区间断层不禁止保存，但要给出警告
- 用户提现弹窗需要同时展示：
  - 自然语言规则说明
  - 实时试算结果
- 对未命中任何规则的金额，不允许用户提交提现申请

## Out of Scope

本次不做：

- 拖拽排序
- 高级 JSON / 可视化双模式切换
- 新的提现流程或新的提现通道
- 自动根据规则模板一键生成默认档位的向导
- 针对不同用户组或不同支付通道配置不同手续费规则

## Approaches Considered

### Approach A: 保留 JSON 文本框，仅替换默认示例

做法：

- 后台仍保留原始 JSON 输入框
- 只把示例文案换成新的 4 档规则示例

问题：

- 没有解决运营配置门槛
- 仍然容易写错边界、排序和字段
- 只是“文案优化”，不是“功能优化”

### Approach B: 继续用侧边抽屉或右侧明细区编辑

做法：

- 规则列表保留在左边
- 点击编辑后，在右侧固定区域或抽屉中编辑

优点：

- “列表 + 明细”模式常见，结构稳定

问题：

- 用户已明确不希望继续走这一交互方向
- 对这种配置型场景来说，列表和编辑区来回切换不够顺手
- 批量微调多条规则时上下文割裂

### Approach C: 规则表格 + 行内展开编辑（采用）

做法：

- 后台主界面展示规则表格
- 点某一行 `编辑` 后，在该行下方展开编辑区域
- 保存时由前端转换为后端规则 JSON

优点：

- 配置上下文连贯，适合连续修改多条规则
- 对运营来说更像“编辑一个分段表”
- 不需要暴露底层 JSON 结构
- 与现有后端配置格式可以兼容

代价：

- 前端需要补一层规则表格模型与 JSON 之间的转换
- 表格会变长，需要控制信息密度

## Decision

采用 **Approach C：规则表格 + 行内展开编辑**。

一句话定义本次能力：

- `管理员通过可视化分段表格维护提现手续费规则，系统在保存时做结构化校验并转换为兼容的规则配置，用户侧同步获得更清晰的规则说明与试算反馈。`

## Admin UX

### Page Structure

提现设置页中，`提现手续费规则` 区域改为规则编辑器，不再直接展示 JSON 文本框。

编辑器由三部分组成：

1. 规则表格
2. 顶部操作区
3. 底部校验与试算反馈区

### Top Actions

顶部提供：

- `新增规则`
- `恢复默认示例`（可选，但推荐）

暂不提供：

- `导入 JSON`
- `导出 JSON`

### Rule Table

每行展示：

- 匹配顺序
- 金额区间
- 收费方式摘要
- 启用状态
- 操作

金额区间展示示例：

- `0 < 金额 <= 100`
- `100 < 金额 <= 500`
- `500 < 金额 <= 2000`
- `金额 > 2000`

收费方式摘要示例：

- `固定手续费 5 元`
- `按 3% 收费`
- `按 2% 收费，最低 1 元，最高 10 元`

操作列提供：

- `编辑`
- `删除`
- `上移`
- `下移`

### Inline Row Editor

点击 `编辑` 后，在当前行下面展开编辑区。

字段包括：

- 起始金额
- 结束金额
  - 留空表示无上限
- 收费方式
  - 固定手续费
  - 按比例
- 固定金额 / 费率
- 最低手续费
- 最高手续费
- 启用状态

交互规则：

- 当收费方式为 `固定手续费` 时：
  - 隐藏 `最低手续费`
  - 隐藏 `最高手续费`
- 当收费方式为 `按比例` 时：
  - 显示 `最低手续费`
  - 显示 `最高手续费`

行内编辑区需要提供：

- `保存`
- `取消`

同时只允许展开编辑一条规则，避免多个编辑区同时展开导致混乱。

## User UX

### Withdrawal Apply Modal

用户提现弹窗继续保留现有主流程：

- 提现金额
- 支付宝账号
- 支付宝姓名
- 提交申请

但手续费相关展示优化为两层：

1. `规则说明`
2. `实时试算`

### Rule Explanation

根据后台规则生成自然语言说明，例如：

- `100 元及以下：固定手续费 5 元`
- `100 元以上至 500 元：按 3% 收费`
- `500 元以上至 2000 元：按 2% 收费`
- `2000 元以上：按 1% 收费`

### Preview

当用户输入金额时，显示：

- 申请金额
- 命中规则
- 手续费
- 预计到账

如果用户输入金额未命中任何规则：

- 不允许提交
- 提示：
  - `当前提现金额未命中任何手续费规则，请调整金额或联系管理员`

## Rule Semantics

### Canonical Boundary Model

本次统一规则语义为“左开右闭”：

- 有上限时：`min_amount < amount <= max_amount`
- 无上限时：`amount > min_amount`

为了兼容第一档从 0 开始的常见场景：

- 当 `min_amount = 0` 时，展示和命中语义仍为：
  - `0 < amount <= max_amount`

### Internal Compatibility

底层仍沿用 `WithdrawalFeeRules` 作为持久化配置格式，但系统内部需要补充一层“编辑器模型”与“规则引擎模型”的映射。

原则：

- 管理员看到的是表格化规则
- 保存时前端转换为现有规则配置结构
- 后端继续做最终校验，防止绕过前端写入脏数据

## Validation Rules

### Per-Rule Validation

每条规则必须满足：

- 起始金额 `>= 0`
- 结束金额为空，或严格大于起始金额
- 固定手续费 `>= 0`
- 比例费率在 `0 ~ 100`
- 最低手续费 `>= 0`
- 最高手续费 `>= 0`
- 若填写最高手续费，则 `max_fee >= min_fee`

### Cross-Rule Validation

规则集合必须满足：

- 不允许区间重叠
- 不允许出现多条无上限规则
- 若某条规则无上限，则其后不能再有规则

### Gap Handling

区间断层处理策略：

- 不阻止保存
- 但给出黄色警告

原因：

- 某些业务可能故意不覆盖所有提现金额段
- 运营需要保留有意识留空的能力

### Save Blocking vs Warning

- 阻止保存：
  - 无效数值
  - 重叠区间
  - 多条无上限规则
  - 无上限规则后仍有后续规则
- 警告但允许保存：
  - 区间断层
  - 存在被禁用规则

## Real-Time Feedback

后台编辑器底部增加“即时反馈”模块。

建议展示：

- 结构化校验结果
- 自然语言规则预览
- 样例试算

样例试算示例：

- `50 -> 手续费 5`
- `100 -> 手续费 5`
- `300 -> 手续费 9`
- `1000 -> 手续费 20`
- `3000 -> 手续费 30`

目的：

- 让运营在保存前直观看到规则结果
- 降低“表面配置正确、实际命中错误”的风险

## Backend Architecture

### Existing Files to Change

- `model/user_withdrawal_setting.go`
- `model/user_withdrawal.go`
- `controller/option.go`

### Backend Responsibilities

后端继续负责：

- 解析 `WithdrawalFeeRules`
- 校验规则有效性
- 在用户提交提现时计算命中规则和手续费

后端新增要求：

- 按新的“左开右闭”语义校验和计算规则
- 对重叠区间给出明确错误
- 保持和前端同一套区间判断结果，避免前后端不一致

## Migration and Compatibility

### Storage Compatibility

本次不新增数据库表，不修改提现单结构。

兼容策略：

- 配置存储仍使用 `WithdrawalFeeRules`
- 新编辑器负责把可视化规则转换成兼容结构
- 老配置在页面加载时转换成编辑器可读模型

### Legacy Config Handling

若历史配置不符合新规则语义：

- 页面加载后应尽量解析
- 无法安全转换时，应在编辑器中显示明确错误
- 不允许在“半解析”状态下静默覆盖旧配置

## Error Handling

管理员侧错误提示应尽量直接说明问题，例如：

- `第 2 条规则和第 3 条规则的金额区间发生重叠`
- `第 4 条规则已设置为无上限，后面不能再添加规则`
- `第 1 条规则的结束金额必须大于起始金额`

用户侧错误提示应偏业务化，例如：

- `当前提现金额未命中任何手续费规则，请调整金额或联系管理员`

## Testing

### Frontend Tests

至少覆盖：

- 编辑器表格渲染
- 行内展开与收起
- 固定手续费 / 比例手续费的字段切换
- 规则转换为配置 JSON
- 重叠校验
- 断层警告
- 自然语言规则说明生成
- 提现弹窗试算展示

### Backend Tests

至少覆盖：

- 规则解析
- 左开右闭边界命中
- 重叠校验
- 无上限规则约束
- 未命中规则时的提现失败路径

## Risks

主要风险：

- 前后端对区间语义理解不一致
- 历史配置转换时出现兼容问题
- 行内编辑让表格高度增长，导致操作区拥挤

对应缓解：

- 统一由 helper 层封装边界判断逻辑
- 后端保留最终校验
- 首版先使用 `上移 / 下移` 而不是拖拽，控制交互复杂度

## Rollout Recommendation

建议分两步实施：

1. 先完成后台可视化规则编辑器与后端语义统一
2. 再补充用户提现弹窗中的规则说明与试算优化

如果开发成本允许，也可以在同一批次一起上线，但实现时应保持代码层面的拆分，避免把配置编辑、规则引擎和用户展示耦合在一起。
