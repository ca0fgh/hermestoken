# 两级订阅返佣全部情况说明

## 先记住判断顺序

系统判断返佣，只看 3 步：

1. 先看这笔订阅有没有支付成功。
2. 再看付款用户的第一层邀请人是谁。
3. 如果第一层是 `直推`，才继续往上看有没有 `团队`。

也就是说：

- 付款用户自己是 `普通`、`直推` 还是 `团队`，都不决定返佣入口
- 决定这单怎么返的，始终是第一层邀请人

---

## 所有返佣情况

### 情况 1：订单没支付成功

结果：

- 不返佣

### 情况 2：付款用户没有邀请人

结果：

- 不返佣

### 情况 3：第一层邀请人是 `普通`

结果：

- 不返佣
- 直接结束
- 不允许跳过这个 `普通`，继续往上找别人拿返佣

### 情况 4：第一层邀请人是 `团队`

结果：

- 这单是 `团队直返`
- 只返给这个第一层 `团队`
- 更上面的人全部不参与

### 情况 5：第一层邀请人是 `直推`，但上面没有 `团队`

结果：

- 这单先给第一层 `直推` 一笔 `直推返佣`
- 没有 `团队级差`
- 直接结束

典型链路：

- `直推1 -> 付款用户`
- `直推2 -> 直推1 -> 付款用户`

用户要点：

- 只有离付款用户最近的那个 `直推` 能拿直推返佣
- 更上面的 `直推` 不会再拿第二份返佣

### 情况 6：第一层邀请人是 `直推`，上面命中 1 个 `团队`

结果：

- 第一层 `直推` 拿 `直推返佣`
- 这个上层 `团队` 拿 `团队级差`

典型链路：

- `团队1 -> 直推1 -> 付款用户`

### 情况 7：第一层邀请人是 `直推`，上面命中多个 `团队`

结果：

- 第一层 `直推` 先拿 `直推返佣`
- 命中的多个 `团队` 再一起分 `团队级差`

典型链路：

- `团队2 -> 团队1 -> 直推1 -> 付款用户`

用户要点：

- 不是所有上级都分
- 只有链路里的 `团队` 参与团队级差
- 一般来说，越靠近第一层 `直推` 的 `团队`，分到的越多

### 情况 8：第一层邀请人是 `直推`，但上层链路里混有 `普通` 或 `直推`

结果：

- 第一层 `直推` 仍然先拿 `直推返佣`
- 上层链路里只有 `团队` 参与 `团队级差`
- 上层的 `普通` 和 `直推` 不拿团队返佣

典型链路：

- `团队2 -> 直推2 -> 团队1 -> 直推1 -> 付款用户`

用户要点：

- `直推2` 不拿返佣
- `团队1`、`团队2` 参与团队级差
- 链路越长，越远的 `团队` 一般分到越少

### 情况 9：第一层邀请人是 `团队`，就算上面还有 `直推` 或 `团队`

结果：

- 仍然只算 `团队直返`
- 只返给第一层 `团队`
- 不会继续往上分

典型链路：

- `直推1 -> 团队1 -> 付款用户`
- `团队2 -> 团队1 -> 付款用户`

用户要点：

- 只要第一层已经是 `团队`
- 这单就不会再进入“直推 + 团队级差”那条链

### 情况 10：付款用户自己是 `直推` 或 `团队`

结果：

- 判断规则不变
- 还是先看第一层邀请人是谁

典型链路：

- `团队1 -> 直推1 -> 直推用户`
- `团队1 -> 直推1 -> 团队用户`

用户要点：

- 付款用户自己的身份，不会让这单多返，也不会让这单少返
- 入口规则还是一样

---

## 最后只记住这 6 句话

1. 只有支付成功的订阅订单才会返佣。
2. 系统永远先看第一层邀请人是谁。
3. 第一层是 `普通` 或没有邀请人，这单不返佣。
4. 第一层是 `团队`，只给这个 `团队`，这叫 `团队直返`。
5. 第一层是 `直推`，先给直推返佣，再往上找 `团队` 分 `团队级差`。
6. 不是所有上级都能拿返佣，只有符合条件的节点才会拿。
